  • Test the actual time performance of the existing Feel++ code

  • Identify the potential part that we can improve

  • Modify it to improve the time performance

2. Work around the Mesh

Previously, we use the method createGMSH ( and so GMSH ) to build the mesh associated to our surface reconstrution problem. As shown ahead, it take a lot of time ( \simeq 70 s ). Two possibilties will be explored here :

  • First, a field, named update, of the createGMSH method can be used to determine how the mesh will be built.

  • After this, we want to generate manually our mesh, without createGMSH.

We will also add the condition on the heigth mean, for the moment equals to 0, in order to compare directly our results to the real surface data.

2.1. Mesh Enhancement

2.1.1. Update field

The update field on the createGMSH method allow us to determine what we want the mesh built and what we want it let behind. In our case, the only useful information is the adjacency between nodes and elements, in order to parallelize in the future the code. Things, like informations on elements face, aren’t needed here and can be shelve to gain time.

This field is used like this

_uh = createGMSHMesh( _mesh=new Mesh<Hypercube<2>>,
                      _structured=1,
                      _h=doption(_name="pixelsize"),
                      _desc=domain(_name="polymere",
                                   _xmax=(x.cols()-1)*doption(_name="pixelsize"),
                                   _ymax=(x.rows()-1)*doption(_name="pixelsize")),
                      _update=size_type(MESH_UPDATE_ELEMENTS_ADJACENCY|MESH_NO_UPDATE_MEASURES));

After recompilation and execution, we finally obtain the mesh built time

[Mesh built] Time : 41.2557s

We have gain \simeq 30 s with this mesh beside the initial one.

2.1.2. Manually mesh built

An another way to gain time on mesh built is to manually build it, without using GMSH. We can so direclty build the wanted informations whitout have to read it from a gmsh file. To manage this, we inspire us from methods used to read nodes and elements from GMSH to use these informations into Feel++.

It used like this in the main code :

auto mesh = boost::make_shared<MeshStructured>( x.rows()-1, x.cols()-1,
        doption(_name="pixelsize"),Environment::worldComm() );

mesh->components().reset();
mesh->components().set( size_type(MESH_NO_UPDATE_MEASURES) );
mesh->updateForUse();
Nodes

The main part of the nodes are define like this

int stop = (partId == (procSize-1)) ? M_ny : cx[partId+1];
for( int j = start ; j < stop; ++j )
{
    std::cout << "j = " << j << std::endl;
    ghosts.clear();
    if (j == cx[partId] && j != 0)
    ghosts.push_back(partId-1);
    else if (j == cx[partId+1]-1 && j != M_ny-1)
    ghosts.push_back(partId+1);

    for( int i = 0; i < M_nx; ++i )
 {
        // point
        int ptid = (M_ny)*i+j;
        //std::cout << "ptId = " << ptid << std::endl;
        coords[0]=M_pixelsize*j;
        coords[1]=M_pixelsize*i;
        point_type pt( ptid,
                       coords,
                       i == 0 || i == M_nx-1
                       || j%((M_ny-1)/procSize) == 0 );

        //// Is on boundary ?
        pt.setMarker(1);
        //// Actual rank ID
        pt.setProcessId( partId );
        //// NO GHOST POINTS I SAID
        pt.setProcessIdInPartition( partId );
        pt.setNeighborPartitionIds( ghosts );
        this->addPoint( pt );
    }
}

Nodes are composed of several elements we will describe here :

  • An id, named ptid

  • A pair of coordinates, coords, which refer to its space position

  • A ProcessId equals to the id of the processor the node belongs

  • A ProcessIdInPartition equals to the id of the processor we currently work on

We choose here to define nodes present at internal boundaries, i.e at interfaces between processors, apart, in case of specific actions we could realise on them.

Elements

The elements are then defined

/*
* Generates elements whithout ghosts
*/
tic();
for( int j = cx[partId]; j < cx[partId+1]; ++j )
{
    ghosts.clear();
    if (j == cx[partId] && j != 0)
        ghosts.push_back(partId-1);
    else if (j == cx[partId+1]-1 && j != M_ny-2)
        {
            ghosts.push_back(partId+1);
        }
    for( int i = 0; i < M_nx-1; ++i )
    {
        element_type e;
        int eid = (M_ny-1)*i+j; // GMSH Id

        e.setId( eid ); // eid == gmsh id
        e.setProcessIdInPartition( partId );
        e.setProcessId( partId );
        e.setMarker(1);
        e.setNeighborPartitionIds( ghosts );


        // points definition
        int ptid[4] = { (M_ny)*(i+1)+j, // 0
                        (M_ny)*(i+1)+j+1, // 1
                        (M_ny)*i+j+1, // 2
                        (M_ny)*i+j // 3
                      };

        for( int k = 0; k < 4; ++k )
        {
            e.setPoint( k, this->point( ptid[k] ) );
        }
        e.setOnBoundary( i==0 || i==(M_nx-2)
                         || j%((M_ny-2)/procSize)==0 );
        // true -> id = global elt id
        // false -> id = local elt id
        this->addElement( e, true );
        // e.id() is modified to Feel++ Id
        __idGmshToFeel.insert( std::make_pair(eid,e.id()));
        // e.id() == Feel++ Id
    }
}

First at all we define the possible processors id that can be neighborhood to the ones we want to build. We can obtain this information with the value of j, our horizontal unknown.

After that we build the several part of an element :

  • An id, named eid

  • A ProcessId equals to the id of the processor the node belongs

  • A ProcessIdInPartition equals to the id of the processor we currently work on

  • A set of points, ptid, which refer to nodes of our element. In this case, we work with square ( hypercube in 2D ) so we have four points.

We also fill a map, __idGmshToFeel, that will make the connection between GMSH Id and Feel Id.

2.2. Height mean condition and error comparaison

We would like to add a condition on the height mean of the calculated surface. In fact, without a condition like this, our main equation can have an infinity of solution ( within a constant ). To realize this, we will use _pre and _post fields of the solve method. We can retrieve an example in testsuite/feelalg/test_prepost_solve.cpp.

2.2.1. Definition and use of the condition

As said at the beginning, we want our height mean to be null to approach the data from the real surface. We define then

auto zeromean =  [&Vh,&u]( vector_ptrtype rhs, vector_ptrtype sol )
{
    double m = mean( _range=elements(Vh->mesh()), _expr=idv(u))(0,0);
    u.add( -m );
};

This method calculate the mean on all the elements from our mesh and then subtract it to our solution u. That give us our wanted height mean.

Then, it only remain its application with the help of the solve method

a.solve(_rhs=l,_solution=u,_pre=zeromean,_post=zeromean) ;

2.2.2. Tests and errors

Now that all this work is done, we want to test it and compare our reconstruct surface with the ral one.

First at all, we can export the two surfaces in order to observe them with Paraview, see if they seem to correspond or not.

The difference of height value between our reconstruct surface and the original one is the part that will interest us more. To do this , we substract the two surface and taking the L2 norm.

std::cout << "L2 error norm : " << normL2( _range=elements(mesh), _expr=idv(u)-idv(s) ) << "\n";

u is our calculated solution and s is the real surface data, obtain from a hbf file. The use of the zeromean condition, develop before, will also vary. Then we obtain :

  • without zeromean : L2 error norm : 0.00433435

  • _pre=zeromean : L2 error norm : 0.00433435

  • _post=zeromean : L2 error norm : 0.00150212

  • _pre=zeromean and _post=zeromean : L2 error norm : 0.00150212

3. Parrallelization

In order to improve even more our program, wa want to have the possibility to run it in parallel. We need then to modify our mesh creation to use it in this case.

3.1. Method and first try

For a first case, we want to cut our mesh into equal parts which will be distribute on each processor, just as the following scheme

We decide to cut our mesh following the y axis. To manage this, this is what we used

int procSize = this->worldComm().godSize();
int cx[procSize];
int id;
std::vector<int>::iterator lowProc;

for (int tmpProc=0;tmpProc<procSize;tmpProc++)
    cx[tmpProc]=(tmpProc+1)*M_ny/procSize;

std::vector<int> vx(cx,cx+procSize);

From procSize, the number of processor used, we create an array cx, which will contain the y-axis index boundaries for each processor, and transform it into the vector vx.

lowProc=std::lower_bound(vx.begin(),vx.end(),j);
id = lowProc - vx.begin();
pt.setProcessIdInPartition( id );
pt.setProcessId( id );

With the std::lower_bound function, we can determine from the index j in wich processsor the associated point is. The result is transformed into an int and then use to give its information.

lowProc=std::lower_bound(vx.begin(),vx.end(),j);
id = lowProc - vx.begin();
e.setProcessIdInPartition( id );
e.setProcessId( id );

The exact same method is used for determine the associated processor for each element.

3.2. Ghost elements

As we just explain ahead, we need to define the ghost elements, i.e the elements present near the one at the interface of two different processes, which will allow us to communicate between the different parts.

After some researchs and attempts, and with the help of V.Huber, the code can now be launched. However, it’s still not functionnal.

Let’s explain this in detail, with the case of ghost elements definition present at the right side of the pprcessor. It’s very similar to regular nodes and elements.

First at all, we need one more column of nodes in order to define our ghost elements. We define then "ghost" nodes, like this

ghosts.clear();
ghosts.push_back(partId+1);
for( int i = 0; i < M_nx; ++i )
{
    int ptId = (M_ny)*i+j_sup+1;
    coords[0]=M_pixelsize*j_sup+1;
    coords[1]=M_pixelsize*i;
    point_type pt( ptId, coords,
                   i == 0 || i == M_nx-1 ); // Is on boundary ?
    pt.setProcessIdInPartition( partId );
    pt.setProcessId( -1 );
    pt.setMarker(1);
    //pt.setNeighborPartitionIds( ghosts );

    this->addPoint( pt );
}

Just as with regular nodes, we define them by coordinates, a ProcessId, the id if the proc that it belongs, and a ProcessIdInPartition, the id of the proc it actually belongs.

We can now create our ghost elements

for( int i = 0; i < M_nx-1; ++i )
{
    element_type e;
    int eid = (M_ny-1)*i+j_sup;
    int eidF = i; //int eidF =(cx[partId]-cx[partId-1])*(M_nx-1)+i+M_nx-1 ;
    e.setId( eidF );
    e.setProcessIdInPartition( partId );
    e.setProcessId( partId+1 );
    e.setMarker(1);

    int ptid[4] = { (M_ny)*(i+1)+j_sup, // 0
                    (M_ny)*(i+1)+j_sup+1, // 1
                    (M_ny)*i+j_sup+1, // 2
                    (M_ny)*i+j_sup // 3 };

    for( int k = 0; k < 4; ++k )
        e.setPoint( k, this->point( ptid[k] ) );

    e.setOnBoundary( i==0 || i==(M_nx-2) );
    nbMsgToRecv[partId+1]+=1;
    // true -> id = global elt id
    // false -> id = local elt id
    this->addElement( e, false );
    mapGhostElt.insert( std::make_pair( eid,boost::make_tuple( eidF, partId+1 )));
    __idGmshToFeel.insert( std::make_pair(eid,eidF));
 }

We can denote here the difference between the value of ProcessIdInPartition, that is equals to the id of the current processor, and ProcessId, which represent the id of the real element. As we explain before, we look here at the right, so ProcessId is the next processor to the one we are.

We also complete two different maps, mapGhostElt and __idGmshToFeel.

  • The first one made the connection between the Gmsh Id of our new ghost element, and it’s Feelpp id in it’s real processor with the id of this one.

  • The second one build the link between the Gmsh id and the Feel id for this element.

This two will be used in the method updateGhostCellInfoByUsingNonBlockingComm, which will update the data on the ghost elements we declared from real elements.

updateGhostCellInfoByUsingNonBlockingComm(this,
                                          __idGmshToFeel,
                                          mapGhostElt,
                                          nbMsgToRecv);
